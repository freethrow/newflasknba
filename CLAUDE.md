# CLAUDE.md — NBA Predictions Modernization Guide

This document instructs Claude (and any developer) on how to modernize the legacy `nba_predictions` Flask app. The current codebase is **Python 2**, Flask 1.0.2 era, with deprecated libraries throughout. The goal is a clean, maintainable rewrite that preserves the existing data and core feature set while adding a real admin workflow and a modern UI.

---

## 1. Current State (what exists today)

**Stack:** Python 2 · Flask 1.0.2 · Flask-SQLAlchemy 2.3 · Flask-Login 0.4 · Flask-Admin 1.5 · Flask-Bootstrap4 · Flask-Script · Flask-WTF · SQLite (`nba.sqlite`).

**Layout:**

```
__init__.py          # app factory, blueprint registration, Flask-Admin setup
config.py            # config classes
models.py            # User, Series, Prediction, Comment + calculate_score()
manage.py            # Flask-Script CLI (deprecated)
auth/                # auth blueprint (login/register/logout)
main/                # main blueprint (home, predictions, leaderboard)
templates/           # Jinja2 + Bootstrap 4 templates
static/              # JS + CSS assets
nba.sqlite           # production data — DO NOT LOSE
reqs.txt, requirements.txt
__init__.pyc, *.pyc  # committed bytecode (delete)
```

**Domain model (must be preserved):**
- `User` — username, password_hash, `is_admin`, `score`. Has a `total` property that sums `score_made` across the user's predictions.
- `Series` — `home`, `away`, `open` (bool), `result` (string like `"4:2"`).
- `Prediction` — FK to `Series` and `User`, `predicted` string, `score_made` int. Unique per `(series_id, user_id)`.
- `Comment` — body + user.
- `calculate_score(pred, actual)` — scoring rules: **15 points** if winner side matches, **10/5/2** bonus based on how close the loser's game count is, **+1** if away team wins.

**Known broken/deprecated things in the current code (must be fixed during the rewrite):**
- Python 2 syntax: `print diff`, `print "..."`, `unicode(self.username)`.
- `current_user.is_authenticated()` is called as a method — in modern Flask-Login it's a property.
- `itsdangerous.TimedJSONWebSignatureSerializer` was removed in itsdangerous 2.x.
- `Flask-Script` is unmaintained; replace with Flask's built-in CLI.
- `Flask-Bootstrap4` is dead; we are removing it entirely (see UI section).
- `slugify==0.0.1` — wrong package, should be `python-slugify` if needed at all.
- `forgery_py` (used in `User.generate_fake`) is Python-2 only.
- `.pyc` files committed to the repo.

## 2. Target State

**Stack:**
- **Python 3.12+**
- **Flask 3.x** (latest stable, currently 3.1.x — verify at install time)
- **Flask-SQLAlchemy 3.x** with the new declarative `db.Model` typed style
- **Flask-Login 0.6.x** (still the right choice — see "Login system" below)
- **Flask-WTF 1.x** for forms + CSRF
- **Flask-Migrate** (Alembic) for schema migrations
- **Argon2** password hashing via `argon2-cffi` (replace bare `werkzeug.security`)
- **HTMX 2.x** for dynamic interactions (no SPA, no React)
- **Tailwind CSS 4.x** via the standalone CLI (no Node build step required, but Node is fine if preferred)
- **uv** or **pip + venv** for dependency management — Marko's preference is `uv`
- **pytest** for tests
- **Ruff** for linting/formatting

**Keep:** SQLite for dev, the existing `nba.sqlite` data, the scoring algorithm, the blueprint structure (`auth`, `main`, plus a new `admin` blueprint).

**Drop:** Flask-Bootstrap4, Flask-Script, Flask-Admin, `itsdangerous` JWT serializer usage, all `.pyc` files, `reqs.txt` (keep only `requirements.txt` or `pyproject.toml`).

## 4. Modernization Plan — Phased

Do these phases **in order**. Each phase should leave the app runnable.

### Phase 0 — Repo hygiene
- [ ] Add `.gitignore` excluding `*.pyc`, `__pycache__/`, `.venv/`, `instance/`, `.env`, `node_modules/`.
- [ ] `git rm` all committed `.pyc` files.
- [ ] Move `nba.sqlite` to `instance/nba.sqlite`. **Back it up first.**
- [ ] Delete `reqs.txt` (keep only one dep file).
- [ ] Create a feature branch: `git checkout -b modernize`.

### Phase 1 — Python 3 + Flask 3 port (no feature changes)
Goal: same app, same UI, but runs on Python 3.12 + Flask 3.x.

- [ ] Create `pyproject.toml` with `uv init`. Pin: `flask>=3.1`, `flask-sqlalchemy>=3.1`, `flask-login>=0.6.3`, `flask-wtf>=1.2`, `flask-migrate>=4`, `argon2-cffi`, `python-dotenv`.
- [ ] Convert all `print` statements to `print()` calls. Remove all `unicode()` calls.
- [ ] Fix `User.is_admin` check: change `current_user.is_authenticated()` to `current_user.is_authenticated` (property, not method).
- [ ] Remove `from itsdangerous import TimedJSONWebSignatureSerializer`. If token-based password reset is needed later, use `itsdangerous.URLSafeTimedSerializer` instead.
- [ ] Replace `Flask-Script`'s `manage.py` with Flask's built-in CLI. Use `flask --app src.nba_predictions run` and custom commands via `@app.cli.command()`.
- [ ] Replace `Flask-Migrate`'s old `MigrateCommand` import.
- [ ] Switch `models.py` to the new SQLAlchemy 2.0 typed style:
  ```python
  from sqlalchemy.orm import Mapped, mapped_column
  class User(UserMixin, db.Model):
      __tablename__ = "users"
      id: Mapped[int] = mapped_column(primary_key=True)
      username: Mapped[str] = mapped_column(String(64), unique=True, index=True)
      ...
  ```
- [ ] Add a `season` column to `Series` (string, indexed, default = `CURRENT_SEASON` env var). Backfill existing rows with `"2019"` or whatever year matches the original data — confirm by inspecting `nba.sqlite` before the migration.
- [ ] Move `User.generate_fake` to a CLI command and replace `forgery_py` with the `Faker` library.
- [ ] Update password hashing: store argon2 hashes via `argon2.PasswordHasher`. Add a one-time migration that re-hashes on next successful login if the existing hash is the legacy werkzeug format (detect by prefix).
- [ ] Run `flask db init`, `flask db migrate -m "initial"`, `flask db upgrade` against a copy of `nba.sqlite`. Verify all rows survive.
- [ ] Smoke-test: register, log in, view leaderboard, submit a prediction.

**Acceptance:** the app runs on Python 3.12 + Flask 3.x with all existing pages working, no template changes yet.

### Phase 2 — Authentication overhaul
Goal: a "better login system" — keep it Flask-native, but secure and modern.

The user asked for a "better login system." We are **not** introducing OAuth/SSO/Auth0 unless asked — that's overkill for a friend-group prediction app. "Better" here means: stronger hashing, CSRF on every form, rate limiting on login, password reset via email token, optional remember-me, and clean session handling.

- [ ] Use `argon2-cffi` for password hashing.
- [ ] Enforce CSRF globally: `CSRFProtect(app)`. All forms render `{{ form.csrf_token }}`.
- [ ] Add `flask-limiter` and rate-limit `/auth/login` to e.g. 5/min per IP.
- [ ] Add password reset flow using `itsdangerous.URLSafeTimedSerializer` (1-hour expiry). Send the email via the **Resend** Python SDK (`resend` package) wrapped in a small `mail.py` helper. Config: `RESEND_API_KEY`, `MAIL_FROM` env vars.
- [ ] Set secure session cookies in production config: `SESSION_COOKIE_SECURE=True`, `SESSION_COOKIE_HTTPONLY=True`, `SESSION_COOKIE_SAMESITE='Lax'`.
- [ ] Add an `admin_required` decorator (don't reuse Flask-Admin's broken pattern):
  ```python
  from functools import wraps
  from flask import abort
  from flask_login import current_user

  def admin_required(f):
      @wraps(f)
      def wrapper(*args, **kwargs):
          if not current_user.is_authenticated or not current_user.is_admin:
              abort(403)
          return f(*args, **kwargs)
      return wrapper
  ```
- [ ] Add a `User.is_active` property (Flask-Login expects it) and an `active` DB column so admins can disable accounts without deleting them.
- [ ] Add tests for login, logout, registration, password reset, and admin-only route protection.

**Acceptance:** all auth flows work, login is rate-limited, passwords are argon2, an admin can be promoted via `flask user promote <username>` CLI command.

### Phase 3 — Custom admin view (replaces Flask-Admin)
Goal: a single admin page where Marko can quickly edit anyone's predictions, results, and series state. Flask-Admin's auto-CRUD is clunky for this use case — we want a purpose-built screen.

- [ ] New blueprint `admin/` mounted at `/admin`, protected by `@admin_required` on every route.
- [ ] **Admin dashboard** (`/admin`):
  - Series table: list all `Series` with `home`, `away`, `result`, `open` toggle. Inline edit via HTMX (`hx-patch` to `/admin/series/<id>`) — change result, flip open/closed, no page reload.
  - Users table: username, total score, is_admin, active. Promote/demote and activate/deactivate inline.
- [ ] **Per-series prediction editor** (`/admin/series/<id>/predictions`):
  - Lists every user's prediction for that series in a table.
  - Each row is editable inline (HTMX): change `predicted`, override `score_made`, save without reload.
  - "Recalculate scores for this series" button — runs `calculate_score()` for every prediction against the current `Series.result` and updates `score_made` in bulk.
- [ ] **Per-user prediction editor** (`/admin/users/<id>/predictions`):
  - Same idea but pivoted by user. Useful when a user complains about a specific score.
- [ ] **Bulk recalculate** CLI command: `flask scores recalculate` — recomputes every prediction's `score_made` and every user's denormalized `score`. Use this after fixing a bug in `calculate_score`.
- [ ] All write endpoints return HTMX-friendly partial templates (`partials/series_row.html`, `partials/prediction_row.html`) so the row swaps in place.
- [ ] Audit log: every admin write appends to an `AdminLog` table (`who`, `when`, `action`, `target_id`, `before`, `after`). Render the last 50 entries at `/admin/log`.

**Acceptance:** Marko can fix any user's prediction in under 5 seconds, flip a series open/closed, edit a result, and see the leaderboard update — all without leaving `/admin` and without full page reloads.

### Phase 4 — Tailwind UI rewrite
Goal: replace Bootstrap 4 with Tailwind 4. Remove Flask-Bootstrap4 entirely.

- [ ] Remove `Flask-Bootstrap4` from dependencies and `bootstrap.init_app(app)` from the factory.
- [ ] Strip all `{% extends "bootstrap/base.html" %}` and `{{ wtf.quick_form(form) }}` calls from templates. Write forms by hand — they're short.
- [ ] Install Tailwind via the standalone CLI (no Node required):
  ```bash
  curl -sLO https://github.com/tailwindlabs/tailwindcss/releases/latest/download/tailwindcss-linux-x64
  chmod +x tailwindcss-linux-x64 && mv tailwindcss-linux-x64 ./tailwindcss
  ./tailwindcss -i src/nba_predictions/static/css/input.css -o src/nba_predictions/static/css/output.css --watch
  ```
- [ ] Build a `base.html` with: a sticky header (logo + nav + user menu), a flash-message region, a `{% block content %}`, and a footer. Include HTMX from a single `<script>` tag in `base.html`.
- [ ] Design tokens — pick a small palette and stick to it. Suggested:
  - Background: `bg-white` / `dark:bg-neutral-950`.
  - Accent: NBA-ish orange `#ED5C2E` mapped to a Tailwind custom color `nba`.
  - Type: `font-sans` (Inter) for body, `font-mono` for scores.
- [ ] **Dark mode toggle.** Configure Tailwind with `darkMode: 'class'`. Add a small inline script in `<head>` of `base.html` that reads `localStorage.theme` (or system preference on first visit) and adds the `dark` class to `<html>` *before* paint to avoid a flash. Toggle button in the header writes back to `localStorage`.
- [ ] Pages to rebuild:
  - **Home / leaderboard** — ranked table of users with total score, medal icons for top 3, season switcher dropdown.
  - **Series list** — card grid, each card shows the matchup, status badge (open/closed), the user's own prediction (if any), and the actual result if finished.
  - **Series detail** — single-series page with prediction form, all users' predictions (revealed once closed), and the comments section.
  - **Prediction form** — inline on the series card. Submit via HTMX `hx-post`, swap the card body with the saved state on success. Validation errors render inline.
  - **My predictions** — table of the current user's predictions with score breakdown.
  - **Login / register / password reset** — centered card layout.
  - **Admin** — see Phase 3.
- [ ] Mobile-first. Test on a 375px viewport — this app is going to be opened on phones during games.
- [ ] Add a 404 and 500 page that match the new design.

**Acceptance:** no Bootstrap classes remain in any template (`grep -r "btn btn-" templates/` returns nothing), Tailwind output CSS is under 30 KB gzipped, every page works on mobile.

### Phase 5 — HTMX integrations
HTMX should be used **only where it pays off** — not everywhere. Targets:

- [ ] **Prediction submit** on the series card — `hx-post` + swap card body. No full page reload.
- [ ] **Admin inline edits** (Phase 3 already covers this).
- [ ] **Leaderboard live refresh** — `hx-get="/leaderboard/fragment" hx-trigger="every 30s"` during open series. Rate-limit the endpoint.
- [ ] **Series open/closed toggle** in admin.
- [ ] **Prediction validation** — validate the `"X:Y"` format on blur via `hx-post="/validate/prediction"`.
- [ ] **Comments** — post a comment on a series page via `hx-post`, swap the comment list partial. Admin delete button on each comment uses `hx-delete` + `hx-confirm`.

For each HTMX endpoint, return a server-rendered partial template, never JSON. Keep the partial templates in `templates/partials/` and reuse them inside the full-page templates so initial render and HTMX swaps share the same markup.

### Phase 6 — Tests, CI, deploy
- [ ] `pytest` covering: scoring algorithm (table-driven), auth flows, admin permission boundaries, prediction CRUD, comments, HTMX endpoints returning partials.
- [ ] GitHub Actions: run `ruff check`, `ruff format --check`, and `pytest` on push.
- [ ] `Dockerfile` only (no docker-compose). SQLite file lives on a mounted volume at `/data/nba.sqlite`. App reads `DATABASE_URL=sqlite:////data/nba.sqlite`.
- [ ] Add `gunicorn` to deps; document `gunicorn 'nba_predictions:create_app()'` as the entrypoint.
- [ ] **SQLite backup cron** on the host: `sqlite3 /data/nba.sqlite ".backup /data/backups/nba-$(date +\%F).sqlite"` daily, keep last 30 days. Document this in the README — losing the file = losing the app.

## 5. Conventions for Claude

When working in this repo, Claude should:

1. **Never break existing data.** Always run migrations against a copy of `nba.sqlite` first. The Belgrade friend group's prediction history is the whole point of this app.
2. **Preserve `calculate_score` semantics exactly** — but extract it to `scoring.py` and write a test table covering the original cases (15 / 10 / 5 / 2 / +1) before refactoring.
3. **Plain JavaScript only.** No TypeScript. No build pipeline beyond Tailwind CLI.
4. **No Mongoose-style ORMs or alternative stacks.** SQLAlchemy native, MongoDB is not used here.
5. **No ShadCN, no React.** This is a server-rendered Flask + HTMX + Tailwind app.
6. **Forms are written by hand** in templates — Flask-WTF for validation/CSRF, but no `quick_form` macros.
7. **One blueprint per concern**: `auth`, `main`, `admin`. Don't add a fourth without a reason.
8. **HTMX endpoints return partials**, never JSON, and partials live in `templates/partials/`.
9. **Use `flask --app src.nba_predictions` CLI** for all management tasks. No `manage.py`.
10. **All admin writes are logged** to `AdminLog`.
11. **Italian/Serbian/English** UI strings — current app has Serbian strings (`"Moraš da se uloguješ..."`). Keep Serbian as the default UI language; structure templates so a future i18n pass with `flask-babel` is feasible (don't hardcode strings in JS).

## 7. Confirmed Decisions

- **Email (password reset):** Resend. Use the official Resend Python SDK, not SMTP. API key in `RESEND_API_KEY` env var. From-address in `MAIL_FROM` env var. Wrap the send call in a tiny `mail.py` helper so it's swappable later.
- **Theme:** Light/dark toggle, persisted in `localStorage`. Use Tailwind's `dark:` variant with `darkMode: 'class'` and a tiny vanilla-JS toggle in `base.html`. Default to system preference on first visit.
- **Leaderboard scope:** All-time per season — leaderboard sums `score_made` across every prediction in the current season. Add a `season` column to `Series` (string like `"2026"`) so future seasons don't pollute the standings. Default season is configurable via `CURRENT_SEASON` env var, with a season switcher in the leaderboard UI for browsing past years.
- **Closing predictions:** Manual only. No scheduler, no auto-close. Admin flips `Series.open` from the admin dashboard. Document this clearly in the admin UI so it doesn't get forgotten.
- **Database:** SQLite in both dev and prod. Keep `instance/nba.sqlite`. Skip the Postgres/Docker-compose work in Phase 6 — just `Dockerfile` + gunicorn + a mounted volume for the SQLite file. Make sure the deploy backs up `nba.sqlite` on a schedule (cron + `sqlite3 .backup`).
- **Comments:** Keep the `Comment` model and wire it up properly. Add a comments section to each `Series` page — users can post short comments, admins can delete. Render via HTMX (post comment → swap comment list partial). No threading, no editing, no markdown — keep it simple chat-style.
